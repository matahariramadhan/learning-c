# ex29_gdb_intro

## Goal

Learn the basic `gdb` workflow by setting a breakpoint, stepping through a simple program, inspecting variables, and entering a function call.

## High-Value Mistake

The most important lesson in this exercise was not a code bug. It was a debugger-timing mistake.

During the `gdb` session, execution stopped here:

```c
int y = 32;
```

Then `print y` showed `0`.

Why that happened:
- `gdb` stops before the highlighted line executes
- at that moment, `y` had not been assigned by `int y = 32;` yet
- the debugger showed a value, but that did not mean `y` was meaningfully initialized

That is the core lesson:
- debugger output depends on the exact execution position
- a variable is not guaranteed to hold its assigned value until the assignment line has actually run

The same thing happened with `result`:
- execution had stopped before `int result = add(x, y);` executed
- so `print result` showed a value before assignment had happened

## Program Used

```c
#include <stdio.h>

int add(int a, int b) {
	return a + b;
}

int main(void) {
	int x = 10;
	int y = 32;
	int result = add(x, y);
	printf("%d\n", result);
	return 0;
}
```

## `gdb` Commands Used

```gdb
break main
run
next
print x
print y
next
print result
step
```

## Observations

- `print x` showed `10` after the line `int x = 10;` had executed
- `print y` was checked while stopped before `int y = 32;` had executed
- `print result` was checked while stopped before the assignment from `add(x, y)` had executed
- `step` entered the `add(a=10, b=32)` function

Useful follow-up command:

```gdb
finish
```

That runs until the current function returns and stops back in the caller.

## Lessons Learned

- `gdb` stops before executing the highlighted source line
- variable values only become reliable after their assignment line has run
- `next` stays in the current function
- `step` enters called functions
- `finish` exits the current function and returns to the caller
